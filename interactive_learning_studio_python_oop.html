Module 1 of 4 · Foundations

Classes & Objects

Every OOP concept builds on this module. We'll learn what a class actually is, how objects come to life from it, and how to keep an object's data safe with encapsulation.

🏗️ What is a class? What is an object?

A class is a blueprint — a description of what data (attributes) and behavior (methods) something should have. An object (or instance) is a real thing built from that blueprint, living in memory with its own actual values.

🍪
AnalogyA cookie cutter is a class: it defines the shape, but it isn't a cookie you can eat. Every cookie you press out is an object — same shape (attributes/methods defined by the class), but each one is a separate piece of dough (its own memory, its own attribute values).
class Dog:
    # class attribute — shared by every Dog
    species = "Canis familiaris"

    def __init__(self, name, breed):
        # instance attributes — unique to each Dog
        self.name = name
        self.breed = breed

    def bark(self):
        return f"{self.name} says Woof!"

rex = Dog("Rex", "Labrador")   # object #1
mia = Dog("Mia", "Poodle")     # object #2
print(rex.bark())   # Rex says Woof!
class Dog (blueprint)
species = "Canis familiaris" (class attribute)
name, breed (instance attributes)
bark()
Click "Instantiate" to see how one blueprint produces many independent objects, each with its own name and breed.

🔐 Encapsulation: protecting an object's data

Encapsulation means bundling data and the methods that operate on it together, and controlling how that data is accessed from outside the class. Python doesn't have true "private" variables like Java — it uses naming conventions instead.

ConventionMeaningExample
publicFreely accessible from anywhereself.name
_protected"Internal use" by convention only — still accessibleself._balance
__privateName-mangled to _ClassName__attr, discourages accidental accessself.__pin

For real validation, Python idiomatically uses @property instead of getter/setter methods:

class BankAccount:
    def __init__(self, balance):
        self._balance = balance

    @property
    def balance(self):
        return self._balance

    @balance.setter
    def balance(self, amount):
        if amount < 0:
            raise ValueError("Balance can't be negative")
        self._balance = amount

⚠️ Common misconceptions
✗ "__init__ is the constructor that creates the object."
✓ Python actually calls __new__ first to create the object; __init__ only initializes it after it already exists.
✗ "Class attributes and instance attributes are the same thing."
✓ Class attributes are shared by every instance; mutating a mutable class attribute (like a list) from one object can silently affect all objects.
✗ "self is a special Python keyword."
✓ self is just a convention — it's the first parameter of any instance method, automatically bound to the calling object.
✅ Key takeaways
  • A class defines structure and behavior; an object is a concrete instance in memory.
  • Instance attributes belong to one object; class attributes are shared across all instances.
  • Python encapsulation is convention-based (_/__) plus @property for real validation.
Module 2 of 4 · Building hierarchies

Inheritance

Inheritance lets a class reuse and extend another class's behavior. This module covers super(), overriding, multiple inheritance, and the Method Resolution Order (MRO) that governs it all.

🌳 Basic inheritance & super()

A subclass inherits every attribute and method from its parent (superclass), and can override or extend them. super() lets a subclass call the parent's version of a method instead of duplicating its logic.

🧬
AnalogyThink of a family tree: a child inherits traits from a parent (eye color, last name) but can also develop their own traits, or override an inherited one (dye their hair a different color). super() is like saying "do what Mom does, then add my own thing."
class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return f"{self.name} makes a sound"

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)   # reuse Animal's init
        self.breed = breed
    def speak(self):              # override
        return f"{self.name} barks!"
One parent, three subclasses — each inherits name but overrides speak() with its own behavior.

🧭 Multiple inheritance & MRO

Python allows a class to inherit from more than one parent. When two parents define the same method, Python decides which one wins using the Method Resolution Order (MRO) — computed with the C3 linearization algorithm. You can always inspect it with ClassName.__mro__.

class Swimmer:
    def move(self): return "swims"

class Walker:
    def move(self): return "walks"

class Duck(Swimmer, Walker):
    pass

print(Duck().move())        # "swims" — leftmost parent wins
print(Duck.__mro__)        # (Duck, Swimmer, Walker, object)

This is why the classic "diamond problem" (two parents that share a common grandparent) doesn't break Python: the MRO guarantees a single, consistent, linear order — each class in the chain is visited exactly once.

✏️ Exercise — predict the MRO

Given class C(A, B): pass where both A and B define greet(), which class's greet() runs when you call C().greet()?

⚠️ Common misconceptions
✗ "Inheritance is always the right tool for code reuse."
✓ Deep inheritance chains get fragile fast. If the relationship isn't truly "is-a", composition is usually safer (see Module 4).
✗ "I can skip super().__init__() if I define my own __init__."
✓ Skipping it means the parent's setup code never runs — often leaving required attributes missing.
✅ Key takeaways
  • super() calls the parent implementation without hardcoding the parent's name.
  • Multiple inheritance is resolved by a predictable MRO (leftmost-first, each class visited once).
  • Prefer shallow, purposeful hierarchies over deep ones.
Module 3 of 4 · Practical application

Polymorphism & Dunder Methods

Polymorphism means "many forms" — the same method call behaves differently depending on the object. Combined with Python's dunder (double-underscore) methods, this is how Python objects integrate naturally with built-in syntax like +, ==, and print().

🎭 Polymorphism & duck typing

Try the demo below: three different classes each implement speak(). The calling code doesn't care which class it's dealing with — it just calls .speak() and trusts each object to respond correctly. That's polymorphism in action.

🎮
AnalogyA universal remote has one "power" button, but pressing it produces a different result on a TV, a soundbar, or a projector. The button (method call) is the same; the behavior is device-specific (object-specific).
>>> Dog().speak() "Woof!"

Python favors duck typing: "if it walks like a duck and quacks like a duck, treat it like a duck." You don't need a shared parent class at all — any object with a compatible method works.

def make_it_speak(thing):
    return thing.speak()   # works for ANY object with .speak()

class Robot:                # no shared base class needed!
    def speak(self): return "BEEP BOOP"

make_it_speak(Robot())      # "BEEP BOOP" — duck typing works

✨ Dunder methods: hooking into Python's syntax

Dunder methods let your objects work with Python's built-in operators and functions. Define them once, and your class behaves like a native type.

DunderTriggered byTypical use
__str__print(obj), str(obj)Friendly, human-readable text
__repr__REPL, debugging, repr(obj)Unambiguous, developer-facing text
__eq__obj1 == obj2Custom equality logic
__add__obj1 + obj2Operator overloading
__len__len(obj)Custom "size" semantics
class Vector:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)
    def __str__(self):
        return f"Vector({self.x}, {self.y})"

v = Vector(1, 2) + Vector(3, 4)
print(v)   # Vector(4, 6) — thanks to __str__ + __add__
✏️ Exercise — implement __eq__

Write the __eq__ method for a Point(x, y) class so two points are equal when their coordinates match.

Model answer:
def __eq__(self, other):
  if not isinstance(other, Point): return NotImplemented
  return self.x == other.x and self.y == other.y
⚠️ Common misconceptions
✗ "__str__ and __repr__ are interchangeable."
✓ __repr__ should be unambiguous (ideally re-creates the object); __str__ is for readable display. If only one is defined, Python falls back to __repr__.
✗ "Polymorphism requires inheritance."
✓ Duck typing gives you polymorphism with zero shared base class — Python cares about behavior, not lineage.
✅ Key takeaways
  • Polymorphism = same call, behavior determined by the object's own type.
  • Duck typing means shared behavior matters more than shared ancestry.
  • Dunder methods integrate your classes with Python's native operators and functions.
Module 4 of 4 · Mastery

Composition, Class/Static Methods & Design Patterns

The final module is about judgment: when to reach for inheritance versus composition, how to use classmethods/staticmethods correctly, and a first taste of design patterns that professional codebases lean on.

🧱 Composition over inheritance

Inheritance models "is-a" relationships. Composition models "has-a" relationships — an object holds a reference to another object and delegates work to it. Composition is usually more flexible because you can swap parts without rewriting a class hierarchy.

🚗
AnalogyA Car is not an Engine — but a Car has an Engine. Modeling "Car(Engine)" would be wrong (inheritance misuse); modeling "Car has self.engine = Engine()" is composition, and lets you swap in an ElectricEngine without touching Car's class definition.

❌ Inheritance misuse ("is-a" violated)

class Car(Engine):   # a Car is NOT an Engine
    pass

✅ Composition ("has-a")

class Car:
    def __init__(self, engine):
        self.engine = engine   # delegate
    def start(self):
        return self.engine.ignite()
AspectInheritance ("is-a")Composition ("has-a")
CouplingTight — subclass depends on parent internalsLoose — depends only on an interface
FlexibilityFixed at class definition timeSwappable at runtime
Best forTrue hierarchical taxonomiesMost everyday "component" relationships

⚙️ classmethod, staticmethod & mixins

class Pizza:
    def __init__(self, toppings):
        self.toppings = toppings

    @classmethod
    def margherita(cls):        # alternate constructor
        return cls(["tomato", "mozzarella"])

    @staticmethod
    def is_vegetarian(topping):   # no access to self/cls needed
        return topping not in ("pepperoni", "ham")

p = Pizza.margherita()

A classmethod receives the class itself (cls) — great for alternate constructors. A staticmethod receives neither self nor cls — it's just a related function namespaced inside the class. A mixin is a small class meant only to be combined with others via multiple inheritance, adding one focused capability (e.g. JSONSerializableMixin).

✏️ Exercise — classmethod or staticmethod?

You need a method Pizza.from_string("tomato,cheese") that parses a string and builds a Pizza. Which decorator fits, and why?

⚠️ Common misconceptions
✗ "Mixins and multiple inheritance are the same thing."
✓ Mixins use the multiple-inheritance mechanism, but by convention they add one narrow capability and are never instantiated on their own.
✗ "staticmethod is basically a classmethod without cls."
✓ Use staticmethod when the logic genuinely doesn't need the class or instance at all — it's really just a grouped utility function.
✅ Key takeaways
  • Prefer composition when the relationship isn't truly hierarchical.
  • classmethod → alternate constructors; staticmethod → grouped utility logic.
  • Mixins add one focused, reusable capability via controlled multiple inheritance.

🧪 Final practical challenge

Design a small notification system that demonstrates everything from this tutorial:

  1. Create an abstract-ish base class Notifier with a method send(message) that subclasses must override.
  2. Create EmailNotifier and SMSNotifier subclasses that override send() with their own formatted output (inheritance + polymorphism).
  3. Give Notifier a __str__ so print(notifier) shows a readable label (dunder methods).
  4. Build a NotificationCenter class that holds a list of notifiers and calls send() on each one — this should be composition, not inheritance.
  5. Add a classmethod NotificationCenter.default_setup() that returns a center pre-loaded with one Email and one SMS notifier.
🚀 Your solution
Reference solution: 
class Notifier:
    def send(self, message):
        raise NotImplementedError
    def __str__(self):
        return self.__class__.__name__

class EmailNotifier(Notifier):
    def send(self, message):
        return f"[EMAIL] {message}"

class SMSNotifier(Notifier):
    def send(self, message):
        return f"[SMS] {message}"

class NotificationCenter:
    def __init__(self, notifiers):
        self.notifiers = notifiers      # composition
    def broadcast(self, message):
        return [n.send(message) for n in self.notifiers]
    @classmethod
    def default_setup(cls):
        return cls([EmailNotifier(), SMSNotifier()])

📝 Summary notes

  • Classes & objects: a class is a blueprint; objects are independent instances with their own attribute values.
  • Encapsulation: use _/__ conventions and @property to control access and validation.
  • Inheritance: models "is-a"; use super(); MRO resolves multiple inheritance predictably.
  • Polymorphism: same call, type-specific behavior — with or without a shared base class (duck typing).
  • Dunder methods: hook your classes into Python's native operators and functions.
  • Composition: models "has-a"; usually more flexible than deep inheritance chains.
  • classmethod / staticmethod: alternate constructors vs. grouped utility logic.
